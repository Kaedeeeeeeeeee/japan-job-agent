# v0.3 freshness-first job policy

## Product invariant

The product searches for as many new public jobs as the configured daily fetch budget permits. It does not maintain a fixed 10,000-job inventory. Only jobs proven to have been published during the last six calendar months are available to recommendation and promotion flows.

## Date states

| State | Admission | Recommendation | Retention action |
|---|---|---|---|
| Known, cutoff through today | Yes | Yes, subject to matching rules | Re-evaluate daily |
| Known, before cutoff | No | No | Purge content; retain fingerprint |
| Known, future date | No | No | Purge content; retain fingerprint |
| Unknown, first seven days | Quarantine | No | Refresh and retry trustworthy extraction |
| Unknown after seven days | No | No | Purge content; retain fingerprint |
| Conflicting | Quarantine semantics | No | Purge after seven days unless resolved |

The cutoff is inclusive and computed in `Asia/Tokyo`. For example, on 2026-07-17 the cutoff is 2026-01-17.

## Bounded catch-up

`DISCOVERY_BACKFILL_DAYS` applies a stricter, one-run collection window without changing the six-month retention
invariant. A 30-day catch-up on 2026-07-17 accepts explicit publication dates from 2026-06-17 through 2026-07-17.
Unknown publication dates are skipped in this mode instead of entering the normal seven-day enrichment quarantine.
Sitemap `lastmod` may be used only to avoid fetching pages that cannot fall in the requested window; final admission
always uses the publication date parsed from the job detail or public ATS response.

## Daily order

1. Search the newest sitemap/list/API entries and refresh persisted official collections.
2. Parse explicit publication fields and materialize Canonical date facts.
3. Promote only `recent` candidates whose official Company–Source relationship is verified.
4. Run `pnpm freshness:apply` after discovery and promotion.
5. Record `job_freshness_daily_metrics` and retry `raw_object_purge_queue` failures.

The default Linux schedule runs the freshness sweep at 08:30 JST. `pnpm freshness:dry-run` is read-only and reports the exact impact before enforcement.

## Re-fetch prevention

`job_retention_tombstones` stores only source identity fields, publication precision/value when known, and SHA-256 identity/normalized-URL fingerprints. Talentio and Engage discovery check URL fingerprints before fetching detail pages. Ingestion checks both identity and URL fingerprints before persisting observations.
