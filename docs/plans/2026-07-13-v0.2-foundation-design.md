# v0.2 Foundation implementation design

本设计采用已确认的方案 B：以可运行的纵向数据链为交付单位。PR #1 先建立可证明的数据不变量和契约；后续 PR 依次实现 Greenhouse、Parser/Evidence/Lifecycle、Canonical/Profile、Ranking/UI/Cloud。LLM 不参与硬过滤、事实判断或排名。

数据从 Connector 的页级原始响应进入 Orchestrator。Orchestrator 是唯一可确认集合完整性的位置；确认后按稳定来源身份 upsert Source Record，以原始字节 hash 去重 Raw Version。Parser 只读取 Raw Version，产生版本化 Extraction 和字段 Evidence。Canonical Materializer 只消费 Extraction ID，并通过唯一主输入和唯一主来源约束生成可回溯版本。

失败安全优先：非 authoritative 同步只更新 Source Health；异常大量缺失先熔断并产生人工审核，不改变岗位缺失计数；合法缺失必须满足时间间隔和次数。Temporal Activity、同步请求、Raw/Extraction、Outbox 发布和消费者副作用分别有独立幂等键，避免重试被误当作新事实。

验证分为纯领域测试和 PostgreSQL 16 + pgvector 集成测试。主 CI 从空库执行全部编号 migration，再运行多对多关系、多值往返、双 primary 拒绝、重试幂等及两个 Publisher 的 `SKIP LOCKED` 竞争测试。外站 live integration 只进入 nightly，不成为主 CI 的不稳定依赖。
